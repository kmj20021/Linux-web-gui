  터미널은 각 사용자마다 독립된 Docker 컨테이너(webterm:latest)를 생성하는
  구조입니다.

    docker run --rm -i --tty
      -v /home/webterm/<username>:/home/user:rw   ← 유일한 호스트 마운트
      -m 256m                                      ← 메모리 제한
      --cpus 0.5                                   ← CPU 제한
      --network bridge
      webterm:latest /bin/bash --login

  컨테이너 내부 실행 유저: user (UID 1000, 비-루트)

  1. 포크 폭탄 :(){ :|:& };:

  동작 원리: 함수가 자신을 재귀 호출하며 프로세스를 기하급수적으로 생성합니다.

  ┌───────────────┬─────────────────────────────────────────────────────────┐
  │     항목      │                          분석                           │
  ├───────────────┼─────────────────────────────────────────────────────────┤
  │ 메모리 제한   │ 256MB 초과 시 커널 OOM killer가 컨테이너를 강제 종료 →  │
  │ -m 256m       │ 자연 소멸                                               │
  ├───────────────┼─────────────────────────────────────────────────────────┤
  │ CPU 제한      │ 프로세스 생성 속도를 늦추지만 근본적 방어는 아님        │
  │ --cpus 0.5    │                                                         │
  ├───────────────┼─────────────────────────────────────────────────────────┤
  │ --pids-limit  │ 가장 큰 취약점. PID 개수 제한 없음. 컨테이너는 호스트   │
  │ 미설정        │ PID 네임스페이스를 공유하지 않지만, 호스트 커널의       │
  │               │ /proc/sys/kernel/pid_max 한계까지 PID를 소모할 수 있음  │
  ├───────────────┼─────────────────────────────────────────────────────────┤
  │ --rm 플래그   │ 컨테이너 종료 시 자동 삭제 → 재연결이면 완전 복구       │
  ├───────────────┼─────────────────────────────────────────────────────────┤
  │ 호스트 영향   │ PID 고갈 상태가 되면 다른 사용자의 컨테이너 생성 실패   │
  │               │ 가능. OOM이 발생하기 전 짧은 시간 동안 호스트 부하 발생 │
  └───────────────┴─────────────────────────────────────────────────────────┘

  결론: OOM에 의해 컨테이너가 종료된 후 재연결하면 완전 복구됩니다. 다만
  --pids-limit이 없어 동시 다수 사용자 환경에서는 일시적 호스트 PID 고갈
  리스크가 존재합니다. 교육 목적으로는 허용 가능하나, 운영 환경에서는 
  --pids-limit 100 정도 추가를 권장합니다.

  2. chown -R nobody /

  예상: 파일 소유권을 모두 nobody로 변경하여 시스템을 망가뜨릴 것으로 보임.

  실제 동작:

  Linux에서 chown은 CAP_CHOWN 권한을 가진 프로세스(루트)만 실행 가능합니다.
  비-루트 유저는 자신이 소유한 파일조차 다른 사용자로 chown할 수 없습니다.

  # user(UID 1000)로 실행 시 결과:
  chown: changing ownership of '/etc': Operation not permitted
  chown: changing ownership of '/home/user/myfile': Operation not permitted
  # → 모든 파일에서 실패

  ┌──────────────────────────────────────────┬──────────────────────────────┐
  │                   항목                   │             분석             │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ 컨테이너 내 /etc, /bin 등                │ 비-루트이므로 chown 불가.    │
  │                                          │ 에러만 출력                  │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ 마운트된 /home/user (호스트의            │ 동일하게 비-루트이므로 chown │
  │ /home/webterm/<username>)                │  불가                        │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ 호스트 파일시스템 영향                   │ 없음                         │
  ├──────────────────────────────────────────┼──────────────────────────────┤
  │ sudo 가능 여부                           │ Dockerfile.webterm에 sudo    │
  │                                          │ 미설치                       │
  └──────────────────────────────────────────┴──────────────────────────────┘

  결론: 아무런 피해도 발생하지 않습니다. 명령어는 모든 경로에서 "Operation not
  permitted"만 출력하고 종료됩니다. 완전히 안전합니다.

  ---
  3. 루트(/)까지 올라가 모든 폴더 접근

  중요 전제: 컨테이너 내부의 /는 호스트의 /가 아닙니다. webterm:latest 이미지의
  격리된 파일시스템입니다.

  ┌─────────────────┬──────────────────────┬────────────────────────────────┐
  │      경로       │  컨테이너 내 접근    │           실제 내용            │
  │                 │      가능 여부       │                                │
  ├─────────────────┼──────────────────────┼────────────────────────────────┤
  │ /etc/passwd,    │ 읽기 가능            │ 컨테이너 이미지의 파일. 호스트 │
  │ /etc/hosts 등   │ (world-readable)     │  정보 없음                     │
  ├─────────────────┼──────────────────────┼────────────────────────────────┤
  │                 │ 읽기 불가            │ 컨테이너의 shadow. 실제 계정   │
  │ /etc/shadow     │ (-rw-r----- root     │ 없음                           │
  │                 │ shadow)              │                                │
  ├─────────────────┼──────────────────────┼────────────────────────────────┤
  │ /bin, /usr 등   │ 탐색 가능, 쓰기 불가 │ 컨테이너 이미지 내용           │
  ├─────────────────┼──────────────────────┼────────────────────────────────┤
  │ 호스트의 /etc   │ 접근 불가            │ 마운트 없음                    │
  ├─────────────────┼──────────────────────┼────────────────────────────────┤
  │                 │                      │ 호스트                         │
  │ /home/user      │ 읽기/쓰기 가능       │ /home/webterm/<username>에만   │
  │                 │                      │ 매핑                           │
  ├─────────────────┼──────────────────────┼────────────────────────────────┤
  │ 다른 사용자 홈  │ 접근 불가            │ 해당 경로 자체가 컨테이너에    │
  │                 │                      │ 없음                           │
  └─────────────────┴──────────────────────┴────────────────────────────────┘

  네트워크 측면: bridge 네트워크로 실행되어 Docker 내부 네트워크에는 접근
  가능합니다. 백엔드 컨테이너 주소를 알면 내부 API 호출이 이론상 가능합니다.
  (단, 인증 토큰 없이는 보호됨)

  결론: 컨테이너 파일시스템을 탐색하는 것은 교육 목적에 이상적입니다. Linux
  파일시스템 구조(/etc, /proc, /var 등)를 실제로 탐험할 수 있고, 호스트에는
  영향이 없습니다. 세션 재연결도 불필요합니다.

  ┌─────────────┬───────────────────┬──────────────┬───────────────────────┐
  │   명령어    │   호스트 위험도   │  복구 방법   │       교육 가치       │
  ├─────────────┼───────────────────┼──────────────┼───────────────────────┤
  │ 포크 폭탄   │ 낮음~중간 (PID    │ 재연결로     │ 높음 (프로세스/리소스 │
  │             │ 한계 미설정)      │ 완전 복구    │  학습)                │
  ├─────────────┼───────────────────┼──────────────┼───────────────────────┤
  │ chown -R    │ 없음 (비-루트     │ 불필요       │ 높음 (권한 체계 학습) │
  │ nobody /    │ 제한)             │              │                       │
  ├─────────────┼───────────────────┼──────────────┼───────────────────────┤
  │ 루트 탐색   │ 없음 (컨테이너    │ 불필요       │ 매우 높음 (파일시스템 │
  │             │ 격리)             │              │  학습)                │
  └─────────────┴───────────────────┴──────────────┴───────────────────────┘

  현재 구조는 교육 환경으로서 잘 설계되어 있으며, 세 시나리오 모두 호스트
  시스템에 실질적 피해를 주지 않습니다. 유일한 개선 고려 항목은 포크 폭탄에 대한
   --pids-limit 추가입니다.
